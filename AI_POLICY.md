# AI Usage Policy

This document describes the project's policy on the use of AI tools for contributions to
`opentelemetry-go-compile-instrumentation`. It applies to all contributors, human and AI agents
alike.

## The Critical Rule

**You must understand every line of code you submit.**

If you cannot explain what your changes do and how they interact with the broader system without
the aid of AI tools, do not contribute those changes. Using AI to write code is fine. Submitting
AI-generated output that has not been understood, reviewed, and owned by a human is not.

This is not anti-AI. It reflects a deeper principle: the fundamentals of software engineering have
not changed. We still need carefully structured, readable, and maintainable code backed by good
tests. AI tools make it easy to turn off your brain; the quality of any contribution depends on the
quality of the thinking behind it, not the tool used to write it.

## AI is Welcome Here

Many contributors and maintainers in this project use AI tools as part of their workflow. We
welcome that.

Our policy exists not because we oppose AI, but because low-effort, unreviewed AI-generated
contributions create maintenance burden without adding value. The problem is not the tool — it is
the absence of human judgment, rigor, and ownership behind the contribution.

## Responsibilities

AI tools do not change the expectations placed on contributors. All the usual responsibilities
remain, regardless of what tools were used:

- **Ownership**: You are responsible for every line of code in your contribution. AI tooling does
  not transfer or reduce that responsibility. If an AI tool produces code that is incorrect,
  insecure, or does not fit the codebase, that is your responsibility to catch and fix before
  submitting.

- **Rigor**: Self-review before peer review is required. Do not submit code you have not read. Read
  the AI-generated output, understand it, question it, and revise it. A reviewer's time should not
  be spent on work you have not already critically examined yourself.

- **Quality**: The same bar applies to AI-assisted contributions as to hand-written ones. Code
  should be idiomatic, well-structured, covered by meaningful tests, and easy to maintain by others
  who were not involved in its creation.

- **Empathy**: Reviewers and maintainers are humans with limited time. Submitting low-effort,
  unreviewed AI-generated output shifts the burden of understanding and validation onto them. That
  is disrespectful of their time.

## AI Disclosure

We appreciate it when contributors disclose the use of AI tools when a significant part of a commit
was taken from a tool without substantial modification. Disclosure is done through an `Assisted-by`
trailer in the commit message:

```
Assisted-by: GitHub Copilot
Assisted-by: Claude Opus 4
Assisted-by: Cursor with Claude Sonnet 4
```

This helps maintainers calibrate their review and set appropriate expectations. Note that for
AI-assisted code that has been substantially reviewed, reworked, and understood by the author,
disclosure is appreciated but not required — the human author owns the result either way.

Do not use `Co-Authored-By` lines for AI tools. As the Go project notes, AI tools cannot be
co-authors for copyright purposes, and such lines carry almost no information about the actual
degree of AI involvement.

## Interactions on GitHub

Discussions on OpenTelemetry repositories — issue comments, pull request reviews, and any other
communication — are for humans only. Never post AI-generated content directly to issues, PRs, or
discussions without thorough human review and editing. The content must be your own words, even if
AI helped you draft them.
